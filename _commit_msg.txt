Pivot server deploy to local-resolve + SFTP-push; fix CRLF in metafile emit

BloomHost's Pterodactyl egg locks the Startup Command, so we can't run
packwiz-installer-bootstrap on the server itself. Replaces the dead
wrapper-based flow (server-deploy-wrapper, server-approve, /home/container
/wrapper/start.sh) with `server-deploy-to-bloomhost`:

  1. Walk docs/packwiz-server/index.toml.
  2. For each metafile entry, fetch the jar from the [download].url
     into .cache/server-resolved/mods/<filename>.
  3. For each non-metafile entry, copy the self-hosted jar through.
  4. SFTP-walk BloomHost's /mods/ and compute a diff (adds + replaces
     + deletes).
  5. Print the plan, prompt y/N (or accept --yes), execute.

Configs are never touched - they're the server admin's domain (same
effect as the preserve_globs guarantee would have been).

Also includes a bug fix for normalize_metafile_bytes that surfaced
during the first deploy attempt: the force_side regex
(`(?m)^side[ \t]*=[ \t]*['"]([^'"]*)['"][ \t]*$`) didn't tolerate \r
before end-of-line. Files written via Path.write_text on Windows end
with CRLF, so the existing `side = ...` line wasn't matched and the
code fell through to the "insert before first [section]" branch,
producing duplicate `side =` lines (broken TOML). resolve_pack then
couldn't parse the affected metafiles and the deploy diff deleted
10 server-side mods (luckperms, bluemap, spark, chunky, etc.) from
BloomHost without uploading replacements. Fixed by normalizing CRLF
to LF before the regex pass; re-deploy restored the missing mods.

What was removed:
- wrapper/ directory (start.sh, approve-update.sh, bootstrap jar, README)
- cmd_server_deploy_wrapper + cmd_server_approve from cli.py
- deploy_wrapper + approve_version helpers from sftp.py
- SftpDeploy.wrapper_push field + the [packwiz_server.deploy].wrapper_push
  config entry
- The "approve + restart" hint in the publish notify text (now points
  at server-deploy-to-bloomhost instead)

What was added:
- src/prism_sync/deploy.py (resolve + plan + execute)
- sftp_walk helper in sftp.py
- server-deploy-to-bloomhost CLI subcommand (with --yes for batch use)
- deploy-server.bat (convenience double-click wrapper)
- publish-everything.bat updated to mention the new deploy step
  (still doesn't auto-deploy - user-selective)

Server state was verified end-to-end via before/after SFTP snapshots:
BloomHost /mods/ now matches the published pack exactly (109 jars =
111 original - 2 FTB .disabled jars + 1 deer-diary-patches version bump),
configs untouched. Server needs a manual restart via the panel to load
the new mod set.
